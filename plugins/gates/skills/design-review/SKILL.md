# design-review

Гейт «дизайн-ревью до разработки» (методология:
`docs/methodology/2026-07-21-codex-review-gates-phase1-design.md` в репо плагина gates).

Все команды гейта ниже запускаются через раннер целевого репо:
`bash .githooks/gates-run codex_review_gate.py <команда> ...`
(если `.githooks/gates-run` нет — проект не онбординат, сначала `/gates-init`).

## Обычный режим

1. Сформулируй дизайн текстом: задача, подход, файлы, инварианты (`CLAUDE.md`/`AGENTS.md`).
   Нетривиальная логика — сначала `spec-driven-development`.
2. **Обязательно — секция BSAC** (бизнес-сценарии и приёмочные критерии): (а) топ-уровневые
   бизнес-сценарии, которыми проверяем работоспособность; (б) как каждый проверяется
   (тест/наблюдение); (в) **явный список денежных краевых случаев** — что может стоить денег
   и как дизайн это предотвращает/детектит. Без BSAC для изменения денег/актуатора — не двигаться.
3. Отправь дизайн+BSAC на независимое ревью Codex: `/codex:adversarial-review` с фокус-текстом,
   который ЯВНО просит судить полноту сценариев, напр.: «Review this design AND its Business
   Scenarios & Acceptance Criteria. Judge: are the top-level scenarios complete; is EVERY
   money-losing edge case enumerated and prevented; what scenarios/edge cases are missing? Flag
   missing money-loss coverage as critical.» (нативный `/codex:review` НЕ годится —
   он код-ориентирован).
4. `critical`/`high` замечания (в т.ч. пропущенный сценарий/денежный случай) → доработай
   дизайн/BSAC и **повтори ревью**. Останавливай цикл по стоп-правилу ниже (НЕ «до нуля находок»).
   BSAC становится тест-матрицей плана.
5. Поставь маркер (разблокирует правки кода в сессии). **File-режим (рекомендуется, тикет #3
   — даёт защиту от дрейфа):** дизайн лежит в файле (напр. `docs/xxx.md`); отправь на ревью
   ИМЕННО его содержимое; после approve — привяжи маркер к файлу его reviewed-хэшем:
   ```bash
   F=docs/2026-..-design.md
   H=$(shasum -a 256 "$F" | cut -d' ' -f1)   # хэш ОТРЕВЬЮЕННОГО содержимого
   bash .githooks/gates-run codex_review_gate.py write-marker design "codex approved: <кратко>" "$H" --file "$F"
   ```
   Гейт сверит `$H` с текущим файлом; при несовпадении — маркер помечается дрейфом (exit 2).
   **Тикет #2:** гейт также требует, чтобы файл нёс секцию сценариев/BSAC/EARS/приёмочных
   критериев (иначе стаб → exit 2, маркер невалиден). Для действительно простой правки без
   матрицы — `--trivial`, не file-маркер.
   Дальше: любая правка `$F` после пометки → G1 блокирует правки кода до ре-ревью+пере-пометки
   (ловит пост-ревью добавление поверхности — см. §«Маркер покрывает только отревьюенный контент»).
   Несколько дизайнов за сессию — повтори `--file` для каждого (все под дрейф-проверкой).

   **Inline-режим (легаси, без защиты от дрейфа):** `write-marker design "..." "$H"` без `--file`.
   (R1-6b: `design` без НЕпустого hash маркер не примет; аварийно при активном инциденте
   актуатора — не SKIP, а kill-switch проекта, см. методологию ML6.)

## ⛔ Маркер покрывает ТОЛЬКО отревьюенный контент (анти-drift, тикет #3)

В **file-режиме** плагин ЭНФОРСИТ это механически: `has_marker` (G1 на каждой правке кода)
пере-хэширует привязанные дизайн-файлы; правка/удаление файла после пометки → дрейф → G1
блокирует правки кода до ре-ревью. В **inline-режиме** (легаси) проверки нет — только правило
дисциплины ниже. Используй file-режим для нетривиальных дизайнов. Правило (обе режима):

- **Любая правка дизайна ПОСЛЕ `write-marker` = маркер больше НЕ покрывает добавленную
  поверхность.** Дописал секцию/сценарий/подход после пометки → перепрогони ревью на новом
  тексте и перепомечай новым хэшем. «Пост-ревью правка» — это НЕ пройденное ревью, а дыра.
- **Необратимая / destructive / actuator / money-поверхность, добавленная пост-ревью, —
  ОБЯЗАТЕЛЬНЫЙ ре-ревью триггер, без исключений.** Удаление/миграция данных БД, DROP/TRUNCATE,
  массовые внешние записи, изменение порога/пола актуатора: если такая секция появилась после
  закрытия Codex-цикла (в т.ч. на хард-капе «по решению человека») — цикл ПЕРЕОТКРЫВАЕТСЯ на
  этой поверхности. Крупный реворк легитимно сбрасывает счётчик сухих раундов (стоп-политика).
- **Safety-net:** такую поверхность независимо ловит и деплой-гейт (Codex-ревью диффа +
  конституция AGENTS.md, где destructive-операции — blocking). Но это ПОЗЖЕ и на уровне кода;
  design-review ловит её на уровне замысла (тот ли подход, есть ли бэкап/откат) — не пропускай.
- Кладите отревьюенный дизайн в `docs/` (коммит) — тогда пост-ревью дрейф виден в git-диффе.

## Тривиальный режим

```bash
bash .githooks/gates-run codex_review_gate.py write-marker trivial "<причина>"
```
— без Codex, осознанно (пишется в аудит).

## Fail-open

Codex недоступен → дизайн-гейт НЕ блокирует: поставь маркер с пометкой и продолжай; per-turn
stop-gate и деплой-гейт (fail-closed) подстрахуют.

## Когда останавливать цикл Codex-ревью (v2)

«Чистое ревью» — НЕ цель: адверсариальный Codex по security-коду почти не сходится к нулю
(бесконечный niche-хвост). Останавливай по КЛАССУ оставшихся находок. Полное правило —
методология, §«Стоп-политика цикла Codex-ревью (v2)». Кратко:
- **Чини в цикле (пол):** fail-OPEN (гейт пропускает опасное/деньги/обход), корректностный баг,
  дешёвый локальный фикс. Порог severity = `critical/high`.
- **НЕ чини (в реестр остатков `AGENTS.md`):** fail-SAFE (пере-блок безобидного), niche/экзотика,
  стиль/нитпик. **Архитектурное** (нужна смена модели) → логируй и ИСКЛЮЧИ из сходимости → Фаза 2.
- **Стоп при:** 2 подряд раунда без новой находки выше пола; ИЛИ раунд в основном шум; ИЛИ хард-кап
  (~5 раундов интерактивно). Severity ревьюера калибруй сам (инфляция severity — норма).
- После стопа: допиши остатки в `AGENTS.md` (scope), чтобы следующий прогон их не пере-флагал.

## Помни

Маркер привязан к сессии и сбрасывается на SessionStart — проходить раз в сессию до первой
правки кода. Bash-мутации кода (sed -i/git apply/redirect в код-путь) тоже гейтятся.
Код-пути определяет `.codex-gate.yaml`; его отсутствие/битость = строгий режим (всё код).
