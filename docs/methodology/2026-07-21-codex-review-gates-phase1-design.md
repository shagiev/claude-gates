# Codex review gates — Phase 1 (local enforcement)

**Дата:** 2026-07-21. **Статус:** design (ожидает реализации). **Фаза:** 1 из 2.
**Ревизия:** переписано под официальный плагин **OpenAI Codex для Claude Code**
(`openai/codex-plugin-cc`, `codex@1.0.6`), установленный в этой сессии — он даёт
review-движок, схему вердикта и stop-гейт, поэтому самодельный `codex exec`-скрипт не
нужен; наши — только дизайн-гейт (pre-edit) и деплой-гейт (commit-bound).

## Контекст и цель

Более качественный CI/CD: (1) деплой с GitHub после мержа (не локально), (2)
обязательное ревью **независимой моделью (OpenAI Codex)** перед разработкой и перед
деплоем. Полный GitHub-энфорсмент (branch protection + деплой-с-мержа + GHCR) — **Фаза
2**. Здесь — **Фаза 1: локальные Codex-гейты** внутри сессии Claude Code.

Явное ограничение Фазы 1 (принято): хуки/гейты работают только в сессии Claude Code —
принуждают действия Claude, не человека из голого терминала, и обходимы (дисциплина +
перехват на границе инструмента, НЕ «физически нельзя»). Негеймибельность — Фаза 2
(GitHub required checks на merge-коммите).

## Готовая инфраструктура плагина (используем, не переписываем)

- **Review-движок** `codex-companion.mjs` (shared Codex runtime). Auth — **ChatGPT-логин**
  (подписка, verified-аккаунт), НЕ API-ключ → без пооплаты токенов.
- **`/codex:review [--base <ref>] [--scope auto|working-tree|branch] [--wait|--background]`**
  — ревью локального git-состояния, вердикт по схеме. Это наш «ревью по запросу».
- **`/codex:adversarial-review`** — ревью с кастомным фокус-текстом (используем для
  дизайн-ревью, где предмет — текст дизайна, а не git-дифф).
- **Stop-review-gate** (ВКЛючён в этой сессии, `reviewGateEnabled=true`) — `Stop`-хук:
  ревьюит правки кода прошедшего хода Claude, `ALLOW`/`BLOCK`; `BLOCK` не даёт закончить
  ход, пока не поправлено. Только edit-produced ходы. Покрывает «каждый ход с кодом
  ревьюится Codex». `SessionStart`/`SessionEnd` — lifecycle runtime.
- **Схема вердикта** (`schemas/review-output.schema.json`): `verdict: approve|needs-attention`,
  `findings[].severity: critical|high|medium|low`, `confidence: 0..1`, `recommendation`.
  Наш **источник контракта** — свой не изобретаем.
- **`codex:rescue`** — агент делегирования тяжёлых задач Codex'у (полезно, вне гейтов).

## Решения (брейншторм)

- **Гибрид-severity → блок:** deploy-гейт роняет деплой при ≥1 находке
  `severity ∈ {critical, high}` (порог настраиваемый); `medium/low` — совещательные.
- **Fail-policy (split):** deploy fail-**closed** (Codex недоступен → блок); дизайн/план
  fail-**open** (пропуск + громкий лог).
- **Сброс дизайн-маркера:** пер-сессия (SessionStart чистит) + тривиальный escape.
- **Auth:** ChatGPT-логин плагина (подписка). API-ключ не требуется.

## Наши компоненты (то, чего плагин НЕ даёт)

### A. `scripts/codex_verdict.sh` — тонкая обёртка над плагином
Не собственный вызов Codex, а **парсер вердикта** для неинтерактивных гейтов
(хук/Makefile не могут «спросить AskUserQuestion» как `/codex:review`):
- Зовёт `node <plugin>/scripts/codex-companion.mjs review --wait --base <ref> --scope <..>`.
- Парсит вывод по `review-output.schema.json`; блок при `severity∈{critical,high}`.
- Exit: `0`=allow, `2`=block. Ошибка/таймаут/нет вердикта → по fail-policy аргументом
  (`--fail-closed`/`--fail-open`), всегда с логом причины.
- Escape-hatch `CODEX_REVIEW_SKIP=1` → exit 0 + громкий WARNING + строка в
  `logs/codex_review_audit.log` (кто/когда/что пропущено). Осознанный обход guardrail.
- Путь к плагину — из `CLAUDE_PLUGIN_ROOT`/резолв; при отсутствии плагина — fail-policy.

### B. `AGENTS.md` — конституция ревьюера
Codex читает ближайший к файлу `AGENTS.md`. Конденсируем инварианты `CLAUDE.md` под
ревью: actuator-safety, доменные money-инварианты, «сценарии текстом до кода», правила
логирования, «каждый code path — тест», «не глотать исключения». Источник правды —
`CLAUDE.md`; дрейф — в «Открытых вопросах».

### C. Скилл `/design-review` — дизайн-ревью ДО разработки
Путь, который плагин не закрывает (stop-gate смотрит код ПОСЛЕ хода, а не дизайн ДО).
Claude формулирует дизайн → ревью через плагин (`/codex:adversarial-review` фокусом на
тексте дизайна, либо `codex_verdict.sh` по дизайн-доку) → при отсутствии
`critical/high` пишет **маркер** `.claude/.design-approved` (SHA-256 дизайна + ts +
`$CLAUDE_SESSION_ID`). Подкоманда `--trivial "<причина>"` — маркер без Codex + аудит.
**Loop (идея из шаблона 2030ai):** при `critical/high` находках — доработать дизайн и
**повторить ревью** до чистого вердикта, только затем маркер. Для деплой-гейта (§E)
loop встроен структурно: фикс меняет `HEAD-SHA` → `check-reviewed` перегоняет ревью.

### D. Хуки (в `.claude/settings.json`, project-level, коммитятся)

| Гейт | Событие | Matcher | Действие | Fail |
|---|---|---|---|---|
| **G1 дизайн** | `PreToolUse` | `Edit\|Write\|NotebookEdit` по кодовым путям | нет свежего `.design-approved` → **deny** «сначала `/design-review`» | open |
| **reset** | `SessionStart` | — | удалить `.claude/.design-approved` | — |

G1 требует дизайн-ревью перед первой правкой кода через Edit/Write и распознанные
Bash-мутации (plan mode не нужен); произвольный shell (cp/mv/python -c/…) НЕ перекрыт —
остаток ML3 (Фаза 2). Best-effort, не абсолют. Хук лёгкий — проверка файла-маркера,
Codex НЕ зовётся на каждую правку. Per-turn
code-review обеспечивает **stop-gate плагина** (уже включён), поэтому отдельный
код-ревью-хук нам не нужен. NB: G1-хук и stop-gate — оба плагина/Claude уровня; их
`SessionStart` не конфликтует (разные скрипты, оба выполняются).

### E. `make deploy: check-reviewed` — деплой-гейт (commit-bound, best-effort)
Плагин ревьюит на stop и по запросу, но НЕ гейтит `make deploy`. Наш `check-reviewed`
(prerequisite `deploy`, рядом с `check-committed`):
1. Требует **чистое дерево** (R1-3) независимо от `SKIP_COMMIT_CHECK`; `baseline` из
   `.claude/.last-deployed-sha`/`CODEX_DEPLOY_BASELINE`, неизвестен → **fail-closed**
   (R1-2, НЕ дефолт `HEAD~1`); `diff_sha256 = sha256(git diff baseline..HEAD)`.
2. Нет валидной записи `logs/review_ledger/<HEAD-SHA>.json` → гоняет Codex-ревью
   (`codex_review_gate.py`, fail-closed) и при allow пишет запись
   `{head_sha, diff_sha256, baseline_sha, verdict, findings, ts}`.
3. Валидирует: `head_sha==HEAD` && `diff_sha256` совпал && нет blocking. Иначе — **exit≠0,
   деплой НЕ стартует**; при недоступном Codex — аварийный контур ML6 (kill-switch/rollback).
4. Успешный деплой пишет `HEAD` в `.claude/.last-deployed-sha`.

Привязка `HEAD-SHA` + `diff_sha256` одобряет ровно **отслеживаемый закоммиченный дифф** —
best-effort, НЕ полный reviewed≡deployed: gitignored конфиг (`.env`) и rsync-дрейф вне
контроля (остаток ML2, Фаза 2). Блокировка деплоя **не останавливает уже работающий
актуатор** — для этого kill-switch. Допишешь коммит → SHA сменился → ревью заново.

### F. Review-лесенка (порядок перед деплоем)
Дешёвые само-проходы сначала, независимый Codex-гейт последним — Codex ревьюит уже
причёсанный код и тратится только на то, что наши проходы не поймали:
1. **`/simplify`** — качество (переиспользование/упрощение/эффективность), применяет правки.
2. **`/code-review`** — охота на баги нашими силами, чиним найденное.
3. (коммит финального состояния)
4. **Codex-гейт** — `check-reviewed`/`/codex:review` на финальном диффе → деплой.

Шаги 1–2 — **та же модель, что писала код** (НЕ независимы): они снижают шум и объём,
но НЕ заменяют независимый гейт. Независимость обеспечивает только Codex (шаг 4) и
per-turn stop-gate. Лесенка — рекомендованный workflow (документируется в `CLAUDE.md`);
жёстко биндится только шаг 4 (деплой-гейт). Enforce 1–2 отдельными артефактами в Фазе 1
не делаем (дисциплина + stop-gate подстраховывает) — кандидат в Фазу 2, если понадобится.

### G. Бизнес-сценарии и приёмочные критерии (BSAC) — обязательный под-гейт дизайн-ревью

**Методология (постоянная часть дизайн-гейта).** Любой дизайн, идущий в `/design-review`,
ОБЯЗАН содержать секцию BSAC: (1) топ-уровневые бизнес-сценарии — что система должна
гарантировать; (2) как каждый проверяется (тест/наблюдение); (3) **явный список денежных
краевых случаев** — что может стоить денег и как дизайн это предотвращает/детектит. Codex
ревьюит BSAC ДО реализации с прямым промптом: «полны ли сценарии; перечислён и закрыт ли
КАЖДЫЙ денежный краевой случай; что упущено» — пропущенный money-loss = `critical`. BSAC
становится тест-матрицей (сценарии → тесты → код). Это `spec-driven-development` +
`actuator-safety`, поднятые до ревьюемого артефакта.

**BSAC для самих гейтов (этот проект).**

Бизнес-сценарии (что гейты обеспечивают — best-effort, не абсолют; в скобках — где проверяется):
- **BS1 reviewed≈deployed (закоммиченный дифф):** деплой не проходит без Codex-одобрения
  отслеживаемого закоммиченного диффа (blocking → блок; неизвестный baseline → блок; грязное
  дерево → блок). НЕ покрывает gitignored `.env`-конфиг и rsync-дрейф (остаток ML2, Фаза 2).
  (тесты `check_reviewed_*`, `test_check_reviewed_make`)
- **BS2 код только после дизайн-ревью (частые пути):** правка кода не начинается без маркера —
  Edit/Write + эвристика частых Bash-мутаций; произвольный shell (cp/mv/python -c/…) — остаток
  ML3 (Фаза 2). docs свободны. (тесты `gate_edit_*`, `gate_bash_*`)
- **BS3 асимметрия деградации:** Codex недоступен → design fail-open (мышление не стопорится),
  deploy fail-closed (непроверенный код не УХОДИТ; блок деплоя **не останавливает уже
  работающий** актуатор — для этого kill-switch, см. ML6). (тесты `decide_*`, `unknown_baseline`)
- **BS4 обход только осознанный и со следом:** `CODEX_REVIEW_SKIP`/`--trivial` → запись в аудит.
  (тесты `skip_audits`, `trivial`)

Денежные краевые случаи (риск → предотвращение; статус после round 2):
- **ML1 [ЗАКРЫТ]** битый/дрейфнувший вывод Codex → молчаливый пропуск плохого деплоя. →
  **R1-1** строгая валидация + **R1-1b** неизвестная severity = blocking (битый = fail-closed).
- **ML2 [ЧАСТИЧНО]** ревьюится не то, что деплоится. → **R1-2** baseline fail-closed при
  отсутствии + **R1-3** чистое дерево закрывают грязный/SKIP-случай; **остаток (Фаза 2):**
  gitignored `.env`-конфиг вне diff, rsync без `--delete`, протухший кросс-машинный baseline.
- **ML3 [ЧАСТИЧНО]** обход через Bash. → **R1-5** эвристика (sed/git apply/patch/redirect);
  **остаток (Фаза 2):** cp/mv/python -c/git checkout/редакторы — полнота через fs-diff-гейт.
- **ML4 [ЗАКРЫТ]** протухший/чужой маркер. → **R1-6** + **R1-6b** session(из hook)+design-hash,
  `""=="" ` закрыт.
- **ML5 [ЗАКРЫТ]** Codex недоступен на деплое → deploy fail-closed (непроверенный не уходит).
- **ML6 [ДЕНЕЖНЫЙ, закрыт вне гейта]** недоступное ревью при АКТИВНОМ перерасходе продлевает
  потерю. → Codex-НЕзависимый аварийный контур: **СНАЧАЛА kill-switch** (флаг заморозки —
  останавливает актуатор), ЗАТЕМ при необходимости rollback (пока заморожено). Rollback
  БЕЗ freeze НЕ останавливает актуатор. Слепой SKIP под инцидентом — НЕ решение.

Два Codex-прогона плана самостоятельно вскрыли ML1–ML4 и переклассифицировали ML6 —
подтверждение, что артефакт работает; BSAC фиксирует их как приёмочные. Полное закрытие
остатков ML2/ML3 — Фаза 2 (см. «Ревизия R2»).

## Ключевые краевые случаи

1. **Циклическая зависимость G1.** `/design-review` пишет дизайн-док через `Write` — G1
   заблокировал бы. **Решение:** G1 гейтит ТОЛЬКО кодовые пути (`app/**`, `tests/**`,
   `scripts/**`, корневые `*.py`, `Dockerfile`, `docker-compose.yml`,
   `config.yaml`). `docs/**`, `.claude/**`, `AGENTS.md`, `*.md` — БЕЗ гейта.
2. **База диффа для деплой-гейта.** См. §E; `check-committed` уже требует чистое дерево
   → HEAD = деплоится, uncommitted-байтов нет.
3. **Codex недоступен / нет вердикта.** design → allow+лог (fail-open); deploy →
   block+лог (fail-closed). Никогда не «тихо ок».
4. **Тривиальная правка.** `/design-review --trivial "<причина>"` — осознанный маркер
   без Codex + аудит.
5. **Escape-hatch деплоя.** `CODEX_REVIEW_SKIP=1 make deploy` — только осознанно, громкий
   лог + аудит (правило feedback_no_silent_overrides).
6. **Stop-gate уже включён** — при реализации правки будут ревьюиться Codex'ом на каждом
   ходу; это ожидаемо (проверка самого механизма на живой работе).

## Что НЕ входит в Фазу 1 (YAGNI)

- GitHub remote, Actions, branch protection, деплой-с-мержа, GHCR — **Фаза 2**.
- Codex Cloud auto-review (GitHub-app, inline на PR) — опция Фазы 2.
- Разделение `.env`/`.env.deploy` — Фаза 2 (DEPLOY_* → GH secrets).

## Тестирование

- `scripts/codex_verdict.sh` — юнит-тесты парсинга вердикта по схеме (approve /
  critical / high / нет вердикта / таймаут × fail-open/closed), с **замоканным
  `codex-companion.mjs`** (стаб-скрипт на пути, печатает заготовленный JSON). Реального
  Codex в тестах НЕТ (изоляция от внешних сервисов — правило проекта).
- `check-reviewed` — нет записи → гоняет (замоканный) ревью и падает при critical/high;
  запись есть, но `diff_sha256` не совпал → падает; валидная allow → проходит; exit≠0
  реально останавливает `deploy`.
- G1-хук — smoke: нет маркера → Edit по `app/**` deny; маркер есть → allow; правка
  `docs/**` без маркера → allow (обход циклической зависимости).
- Escape-hatch — `CODEX_REVIEW_SKIP=1` → exit 0 + аудит-строка.

## Ревизия R1 — по итогам Codex-ревью плана (2026-07-21)

Codex (adversarial-режим) отревьюил план и нашёл 6 валидных дыр. Фиксы (все приняты):

- **R1-1 (crit): строгая валидация вердикта.** Парсер НЕ должен трактовать битый/дрейфнувший
  вывод как «чисто». `decide_exit` fail-closed'ится, если: вердикт None, ИЛИ вердикт не из
  `{approve, needs-attention}`, ИЛИ `needs-attention` без распарсенных находок (дрейф формата).
  Только явный `approve`/не-blocking = allow. Тесты: пустой вывод, неизвестный вердикт,
  неизвестная severity, усечённые находки, дрейф рендера. (Предпочтительно — JSON по
  `review-output.schema.json`; рендер как фолбэк со строгой проверкой.)
- **R1-2 (high): baseline fail-closed.** Нет `.last-deployed-sha` (свежий клон/другая машина —
  файл в `.claude/`, gitignored) → НЕ дефолтить на `HEAD~1`. Неизвестный baseline = деплой-гейт
  **блокирует** (fail-closed) с требованием задать/подтвердить задеплоенный SHA (в идеале — с
  сервера). Тесты: отсутствующий/протухший/не-предок/кросс-машинный baseline.
- **R1-3 (high): reviewed ≡ deployed на чистом дереве.** `make deploy` шлёт рабочее дерево
  (rsync) и `SKIP_COMMIT_CHECK=1` разрешает грязный деплой. `check-reviewed` **сам** требует
  чистое дерево (`git status --porcelain` пуст) НЕЗАВИСИМО от `SKIP_COMMIT_CHECK`, и хэширует
  `baseline..HEAD` — так задеплоенное дерево == отревьюенный HEAD. Полный артефакт-биндинг
  (деплой неизменяемого архива из отревьюенного коммита) — Фаза 2 (GHCR).
- **R1-4 (high): un-ignore `.claude`.** `.gitignore:42` игнорит весь `.claude/` → хук и скилл
  не коммитятся. Отдельная задача: `!.claude/settings.json` + `!.claude/skills/**` (и родительские
  `!.claude/`), сохранив игнор `.claude/.design-approved`, `.claude/.last-deployed-sha`, `logs/`.
  Проверка `git check-ignore` + `git ls-files`.
- **R1-5 (high): Bash-лазейка G1.** Matcher `Edit|Write|NotebookEdit` не ловит правку кода через
  `Bash` (sed -i / git apply / patch / `> file` / tee). Добавляем `PreToolUse` на `Bash`:
  эвристика — при отсутствии маркера И признаках мутации кодового пути (redirection в код-путь,
  `sed -i`, `git apply`, `patch`, `tee <код-путь>`) → deny. Остаточная лазейка (произвольный
  shell) — задокументирована; полный контроль — Фаза 2 (серверный gate на merge-коммите).
- **R1-6 (med): маркер привязан к session+design.** `write_marker` пишет SHA-256 дизайна +
  `CLAUDE_SESSION_ID`; `has_marker` валидирует: файл есть, сессия совпадает с текущей, поле
  design-hash присутствует (тривиальный режим — поле `trivial`). Атомарная запись. Тесты:
  протухший/чужой-сессии/битый/тривиальный маркер.

## Ревизия R2 — Codex round 2 → честный де-скоп Фазы 1 (2026-07-21)

Второй Codex-прогон показал, что часть R1-фиксов была поверхностной, а часть гарантий
**в принципе недостижима** в модели «rsync рабочего дерева + локальный baseline + свобода
shell». Решение (пользователь, вариант A): дешёвые корректные фиксы берём, остальное —
честно де-скопим, гарантии выносим в Фазу 2. **Прекращаем патчить локальную модель ради
гарантий, которых она не даёт** (правило анти-whack-a-mole).

**Дешёвые корректные фиксы (в Фазе 1):**
- **R1-1b:** `blocking` = critical/high ИЛИ любая НЕизвестная severity (напр. `[urgent]`) —
  чтобы нестандартная метка не проскочила как не-blocking.
- **R1-6b:** маркер валиден только при НЕпустой сессии (из hook-input, не только env) И
  (design→непустой design-hash | trivial); `""=="" ` закрыт; trivial пишется в аудит.
- **ML6 переклассифицирован в денежный:** недоступное 900-сек ревью при АКТИВНОМ перерасходе
  продлевает потерю. Аварийный контур — **Codex-НЕзависимый, в порядке**: СНАЧАЛА kill-switch
  (флаг заморозки — мгновенно глушит актуатор), ЗАТЕМ при необходимости rollback (пока
  заморожено); rollback БЕЗ freeze актуатор НЕ останавливает. Слепой `SKIP` под инцидентом —
  НЕ решение.

**Честно де-скоплено (best-effort в Фазе 1, полная гарантия — Фаза 2):**
- **R1-3 остаток:** `check-reviewed` требует чистое дерево, но rsync шлёт рабочее дерево
  **без `--delete` и включая gitignored `.env`** → изменение актуаторного конфига в `.env`
  деплоится ВНЕ ревью (его нет в `baseline..HEAD`); плюс TOCTOU при правке during-review.
  Полное «reviewed≡deployed» = деплой неизменяемого артефакта из отревьюенного коммита +
  явный биндинг конфига (Фаза 2).
- **R1-2 остаток:** локальный `.last-deployed-sha` может протухнуть (деплой с другой машины) →
  ревьюим не тот диапазон. Авторитетный baseline = задеплоенный SHA **с сервера** (Фаза 2).
- **R1-5 остаток:** Bash-эвристика ловит sed/git apply/patch/tee/redirect, но `cp`/`mv`/
  `python -c`/`perl -pi`/`git checkout`/редакторы — проходят. Полнота = post-command fs-diff
  или read-only allowlist (Фаза 2). **Фаза 1 НЕ заявляет полное перекрытие Bash.**

**Явная позиция:** Фаза 1 = локальная best-effort дисциплина, поднимает планку и ловит частые
случаи; **НЕ заявляет «reviewed≡deployed» и негеймибельность**. Эти гарантии — только Фаза 2.

**Новые сценарии для BSAC Фазы 2** (Codex round 2): artifact drift, конкурентная правка
during-review, кросс-машинный/протухший baseline, degraded-review при активном перерасходе,
провал rollback, отказ записи аудита.

## Стоп-политика цикла Codex-ревью (v2, 2026-07-22)

Проблема (эмпирика этого проекта): адверсариальный Codex по security-гейту почти НЕ сходится
к нулю находок — бесконечный niche-хвост (экзотический shell, layout'ы установки, конкурентные
сессии, TOCTOU-окна). Критерий остановки — по **классу** оставшихся находок, НЕ по их отсутствию.
Совпадает с индустриальным консенсусом (ссылки внизу — мы пришли к тому же независимо).

**Правило:**
- **Пол (чиним в цикле)** — находка, которая: (1) **fail-OPEN** (гейт ошибочно ПРОПУСКАЕТ опасное:
  непроверенный деплой, утечка денег, обход); (2) **корректность** (реальный баг/краш);
  (3) **дешёвый локальный фикс реального дефекта**. Порог severity = `critical/high` (ТОТ ЖЕ, что
  блокирует сам гейт); задаётся в промпте ревьюера.
- **Ниже пола (НЕ чиним, в реестр остатков)** — **fail-SAFE** (пере-блок безобидного — трение, не
  потеря), niche/экзотика с пренебрежимой вероятностью, style/нитпик.
- **Архитектурное (cross-boundary)** — **логируем, но ИСКЛЮЧАЕМ из проверки сходимости**: уходит в
  дизайн Фазы 2 / к человеку, НЕ в заплатку (индустрия дословно: «logged but excluded from the
  convergence check»). Наша «граница Фазы».
- **Остановка при любом из:** (а) 2 подряд «сухих» раунда — нет НОВОЙ находки выше пола;
  (б) раунд преимущественно из шума (низкий signal-ratio: сигнал = runtime-ошибки/уязвимости/
  архитектура/перф, шум = стиль/микро-опт; хороший поток >60% сигнала); (в) хард-кап (≈5 раундов
  интерактивно / 1-2 в CI) — бэкстоп независимо от сходимости.
- **Severity калибруем САМИ** — не доверяем метке ревьюера вслепую (инфляция severity рушит доверие
  быстрее пропусков; в этом сеансе Codex рейтил fail-safe-мелочи как P1/P2).
- **Кросс-модель консистентность** — fail-safe/niche находка из одного прохода = кандидат в шум;
  держим 2 независимые модели (Claude-ревьюеры + Codex).
- **Реестр остатков в `AGENTS.md`** — задокументированный остаток = «известный», ревьюер не должен
  пере-флагать (иначе цикл вечно мусолит одну жилу).

**Определение готово для гейта под адверсариальным ревью:** не «Codex вернул 0 находок», а «нет
находок выше пола, и оставшийся хвост — в реестре остатков + Фаза 2».

## Фаза 2 (эскиз, отдельная спека)

GitHub remote (private) → Actions: на PR тесты+mypy+coverage + **ревью** как required
checks на merge-коммите (негеймибельно); на merge в main — build → push GHCR → сервер
`docker compose pull && up` → healthcheck → автооткат на предыдущий tag. Разделение
секретов, SSH-deploy-key в GH secrets.

**Истинные гарантии, закрывающие остатки R2 (то, что Фаза 1 не может):**
- **reviewed≡deployed** (ML2): деплой **неизменяемого артефакта** (образ GHCR по digest,
  собранный из отревьюенного коммита) + **явный биндинг конфига** (`.env`/actuator-config
  проходит контроль, а не приезжает мимо ревью через rsync); re-check digest перед выкаткой.
- **авторитетный baseline** (ML2): задеплоенный SHA/digest берётся **с сервера**, не из
  локального файла; проверка ancestor/несоответствия.
- **полное перекрытие мутаций** (ML3): гейт на merge-коммите — любой путь правки (Bash,
  генераторы, другая машина) проходит один серверный check; локальные эвристики не нужны.
- BSAC Фазы 2 включает: artifact drift, конкурентную правку, кросс-машинный baseline,
  degraded-review при активном перерасходе, провал rollback, отказ аудита.

**Идеи из шаблона 2030ai (`2030ai-pullrequest-pipeline-skill-template`):**
- **Панель независимых ревьюеров** вместо одного Codex: агрегировать вердикты
  Codex + Copilot + Cursor Bugbot + Claude-review (каждый триггерится PR-комментом).
  Несколько независимых глаз > один; блок при `critical/high` у любого.
- **`reviewers.yaml`-конфиг**: режимы (medium/max) × набор ревьюеров × severity-порог —
  менять состав конфигом, не кодом. Наш `codex_verdict.sh` уже reviewer-agnostic по
  схеме — расширяется до агрегатора нескольких вердиктов.
- **Cost-guard**: дорогое ultra-ревью — только по инициативе пользователя, не из
  agent-shell (как наш billed `/code-review ultra`).
- **Отличие, которое НЕ перенимаем:** их fail-open на отсутствующем ревьюере — у нас
  deploy остаётся fail-closed (актуатор). Их конфиг без severity/merge-правил — у нас
  они есть (commit-bound ledger).
