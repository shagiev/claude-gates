# Enforced review ladder (/simplify → /code-review) — Phase 1.5

**Дата:** 2026-07-22. **Статус:** design (ожидает Codex-ревью + реализации).
**Расширяет:** `2026-07-21-codex-review-gates-phase1-design.md` (§F Review-лесенка).

## Цель

Сделать «review-лесенку» (§F Фазы 1) не рекомендацией, а **проверяемым, commit-bound
шагом**: перед коммитом/деплоем доказать, что `/simplify` и `/code-review` (свои проходы
Claude) реально **выполнены на финальном коде**. Это НЕ замена независимого Codex-гейта:
проходы self-attested (Claude ревьюит свой код), поэтому доказываем «шаг выполнен», а не
«сделан честно» — честность даёт только независимый Codex (stop-gate + деплой-гейт).

Решения (брейншторм): оба прохода обязательны; строгость = маркер «запущено»; энфорсмент
на pre-commit И на деплое; механизм — git-хуки (ловят и терминальные коммиты, не только
Claude-сессию).

## Архитектура

**Поток:** написали код → `/simplify` (правки) → `/code-review` (правки) → **коммит**
(pre-commit проверяет оба маркера) → Codex stop-gate / деплой-гейт.

### 1. Маркеры проходов (tree-chain, R5; двухшаговый протокол R6)
**Протокол `begin`/`mark` (R6, Codex-фикс: `tree_before` нельзя восстановить после мутации —
его надо снять ДО прохода):** перед запуском прохода — `ladder_gate.py begin <pass>` (снимает
`tree_before` в pending-файл), после применённых фиксов — `ladder_gate.py mark <pass>` (снимает
`tree_after`, пишет маркер из pending + after; `mark` без `begin` → ошибка).
**Анти-replay (R7, Codex):** pending — **одноразовый** (`mark` его ПОТРЕБЛЯЕТ/удаляет; повторный
`mark` без нового `begin` → ошибка); `begin` перезаписывает свой pending; `begin code-review`
валидирует старт цепочки — его tree обязан совпасть с `simplify.tree_after` (иначе ошибка
«сначала /simplify»: ручные правки между проходами ловятся сразу, а не на pre-commit).
**Порядок consume-then-publish (R8, Codex):** `mark` сначала УДАЛЯЕТ pending, затем атомарно
(tmp+rename) публикует маркер. Crash между шагами → нет ни pending, ни маркера → повторить
begin/mark (fail-closed, работа не «подтверждается» частично). Итоговый маркер
`.claude/.ladder-simplify` / `.claude/.ladder-codereview` = `{tree_before, tree_after, session, ts}`:
- `tree_*` — git-TREE-хэш состояния «всё изменённое» ДО и ПОСЛЕ прохода. Считается через
  **ВРЕМЕННЫЙ индекс** (`GIT_INDEX_FILE=<tmp-копия> git add -A && git write-tree`) — реальный
  индекс пользователя НЕ мутируется (R5: `git add -A` в реальный индекс — инвазивный side-effect).
  `git diff HEAD` не годится (упускает untracked/staging).
- **Chain-семантика (R5, ключевой фикс):** проходы ПОСЛЕДОВАТЕЛЬНЫ и мутируют код — «оба маркера
  == финальное дерево» невыполнимо (фиксы `/code-review` протухают маркер `/simplify` → вечный
  пинг-понг). Валидная лесенка = ЦЕПОЧКА: `simplify.tree_after == codereview.tree_before` И
  `codereview.tree_after == текущее дерево`. Т.е. code-review видел ровно выход simplify, а
  текущее состояние = ровно выход code-review. Фиксы последнего прохода ревьюит независимый
  Codex-гейт (следующий слой). Ручная правка МЕЖДУ проходами рвёт цепочку → блок → перепрогон.
  Порядок канонический: simplify → code-review.
- Pre-commit (на РЕАЛЬНОМ индексе — см. §2) и скилл-mark (на временном) считают tree одним
  модулем `scripts/ladder_gate.py`.
- Остаток: частичный коммит (`git commit <paths>`) даёт дерево ≠ «всё изменённое» → блок
  (fail-closed, консервативно верно).

### 2. Pre-commit гейт (`.githooks/pre-commit`)
Включается `make setup-hooks` → `git config core.hooksPath .githooks`.
- **Scope (R5): гейтятся только коммиты, трогающие КОД.** Если ни один изменяемый путь коммита
  не проходит `is_code_path` (docs/**, *.md, .claude/** …) → exempt, лесенка не требуется
  (иначе каждый докс-коммит требовал бы двух model-проходов — сам этот спек-коммит был бы
  заблокирован). Переиспользуем `is_code_path` из `codex_review_gate.py`.
- **R6 (Codex): `.githooks/` и `Makefile` добавляются в КОД-пути** (`CODE_PATH_PREFIXES`/`EXACT`
  Фазы 1) — иначе коммит, ослабляющий сам pre-commit-хук или деплой-рецепт, проходил бы как
  «не-код» без лесенки и мимо G1. Остаток: правка `AGENTS.md` (конституция ревьюера) — .md,
  вне гейтов by design (G1-циклическая зависимость); её ослабление видно Codex-гейту в диффе,
  авторитетный pin конституции — Фаза 2.
- Если `ladder.enabled=false` → пропуск (флаг — удобство разработчика; **деплой-гейт всё равно
  требует лесенку**, см. §4). Escape: `LADDER_SKIP=1` → allow + громкий аудит.
- Вычисляет tree РЕАЛЬНОГО индекса (`git write-tree` — на pre-commit это ровно то, что
  закоммитится) и валидирует **цепочку** (§1): `simplify.after == codereview.before` И
  `codereview.after == индекс`. Разрыв/протухание/отсутствие → **abort commit** с инструкцией
  (R6, полный протокол): «`ladder_gate.py begin simplify` → /simplify → `mark simplify` →
  `begin code-review` → /code-review → `mark code-review`». Fail-closed. Работает и для
  терминальных коммитов.

### 3. Post-commit хук (`.githooks/post-commit`)
Пишет `logs/ladder_ledger/<HEAD-SHA>.json`:
- лесенка валидна (`codereview.after == HEAD^{tree}` + цепочка) → `{passes:[simplify,
  code-review], tree, session, ts}`;
- коммит был exempt (не-код) → `{passes:["exempt-noncode"], tree, ts}`;
- коммит прошёл под `LADDER_SKIP=1` → `{skipped:true, reason:$LADDER_SKIP_REASON|"", tree, ts}`
  (R5: иначе один skip-коммит навсегда «отравляет» диапазон — каждый будущий деплой требовал бы
  blanket-skip, отключающий проверку ВСЕХ коммитов).
- NB: post-commit в git не может абортить — провал записи печатает ГРОМКО в stderr; отсутствие
  записи безопасно (fail-closed: деплой-гейт заблокирует и подскажет).

### 4. Деплой-гейт (`check-reviewed`, расширение)
- **Для КАЖДОГО коммита в `baseline..HEAD` валидирует ledger-запись** (Codex-фикс: только-HEAD
  не покрывает непроверенные промежуточные коммиты). Проверка НЕ presence-only (Codex-фикс).
  Коммит диапазона считается покрытым, если выполнено ЛЮБОЕ из:
  1. валидная запись: `ledger[commit].tree == git rev-parse <commit>^{tree}` И присутствуют все
     **КАНОНИЧЕСКИЕ** проходы — константа `DEPLOY_REQUIRED_PASSES` в коде гейта, НЕ мутабельный
     `config.required_passes` (Codex-фикс: иначе коммит урезает config и ретроактивно ослабляет
     весь диапазон; правка константы — код гейта → ревьюит Codex);
  2. exempt-запись `passes:["exempt-noncode"]` с совпавшим tree (не-код коммит, R5);
  3. `skipped:true`-запись с совпавшим tree (R5) — честится с **громким аудитом** в выводе
     деплоя (skip уже был осознанным и записан в аудит на коммите);
  4. **grandfathering (R5):** коммит — предок `LADDER_EPOCH_SHA` (константа в коде гейта = SHA
     внедрения фичи; иначе вся до-ladder история навсегда блокирует первый деплой);
  5. **merge-коммит (>1 родителя) — документированный остаток:** exempt с громкой пометкой
    (хуки не гоняются на merge/rebase-механике; итоговый диапазон целиком ревьюит Codex-гейт).
  Всё прочее (нет записи / tree не совпал / не все канонические проходы) → блок.
- **Не зависит от `ladder.enabled` и `config.required_passes`** (Codex-фикс: иначе коммит флипает
  конфиг и обходит гейт). Отключить деплой-проверку лесенки — только `LADDER_SKIP=1` (аудит).
- Изменения `.githooks/**`, `ladder.*`, констант гейта — обычные кодовые пути → ревьюит **Codex**.
- Порядок в `check_reviewed_cli`: clean-tree → baseline → **ladder-ledger диапазона** →
  Codex-skip/ledger/ревью. **Семантика скипов (R5, явно):** `LADDER_SKIP=1` пропускает ТОЛЬКО
  ladder-проверку (Codex остаётся); `CODEX_REVIEW_SKIP=1` пропускает ТОЛЬКО Codex-часть (ladder
  остаётся); полный обход = оба флага (два осознанных audited-решения).

### 5. Скиллы — НЕ редактируем (R5)
`/simplify` и `/code-review` — ПЛАГИННЫЕ скиллы (файлы вне репо, редактировать нельзя —
первоначальный план «добавить шаг в их конец» нереализуем). Вместо этого **энфорсим исход**:
- после каждого прохода Claude вызывает `python3.13 scripts/ladder_gate.py mark <pass>`
  (конвенция, документируется в `CLAUDE.md`);
- забыл — **pre-commit сам скажет**, что прогнать (self-healing: отказ содержит точную
  последовательность команд). Исход энфорсится, шаг-в-скилле не нужен.

## Конфиг
`ladder.enabled: bool` — **только pre-commit** (удобство разработчика); деплой-гейт требует
лесенку НЕЗАВИСИМО от флага (отключить деплой-проверку — только `LADDER_SKIP=1`, аудит).
`ladder.required_passes: [simplify, code-review]` — какие проходы обязательны **на pre-commit**
(удобство). Деплой-гейт использует КАНОНИЧЕСКУЮ константу кода (`DEPLOY_REQUIRED_PASSES`), НЕ этот
config — чтобы урезание config не ослабляло деплой. В `config.yaml`.

## BSAC — бизнес-сценарии и денежные краевые случаи

Бизнес-сценарии (где проверяется):
- **BS-L1:** коммит невозможен, если хоть один обязательный проход не выполнен на финальном
  коде (pre-commit). (тест: маркер отсутствует/устарел → abort)
- **BS-L2:** деплой невозможен, если для ЛЮБОГО код-коммита диапазона `baseline..HEAD` нет
  валидной ladder/exempt/skipped-записи (деплой-гейт). (тест)
- **BS-L3:** git-хуки не настроены на клоне → коммиты проходят, НО деплой всё равно блок
  (нет ladder-ledger) → fail-closed. (тест)
- **BS-L4:** обход только осознанный (`LADDER_SKIP=1`) и со следом в аудите. (тест)

Денежные краевые случаи (риск → предотвращение):
- **ML-L1:** молчаливый пропуск гигиены → грязный/непроверенный код к Codex/деплою. →
  pre-commit + деплой-ledger (два места).
- **ML-L2:** проход запущен на СТАРОМ коде, потом правки → ложное «пройдено». → content-hash
  сверка (маркер протухает).
- **ML-L3:** коммит мимо Claude (терминал) → git-хук ловит; если хуки не настроены — деплой-гейт.
- **ML-L4:** маркер одной сессии/чужой → в маркере есть session; content-hash — основная
  привязка (правки другой сессии меняют content). 
- **ML-L5** (обратная сторона, НЕ денежная): лесенка тормозит хотфикс при инциденте →
  `LADDER_SKIP=1` (audited), как `CODEX_REVIEW_SKIP`; аварийно — kill-switch проекта.
- **ML-L6:** непроверенный промежуточный коммит в `baseline..HEAD` (грязь без лесенки, затем
  чистый HEAD) → деплой требует ladder-ledger для **каждого** коммита диапазона (§4).
- **ML-L7:** коммит сам ослабляет гейт (`ladder.enabled=false` / урезание `required_passes` /
  правка `.githooks/**`) → деплой требует лесенку НЕЗАВИСИМО от config: канонический набор проходов
  — константа кода (`DEPLOY_REQUIRED_PASSES`), деплой не читает мутабельный config; изменения
  хука/константы ревьюит независимый Codex-гейт.

## Что НЕ делаем (осознанно) + остатки (R5)
- «Проход прошёл честно» — не проверяем (self-attested; независимость = Codex-гейт).
- «Findings addressed» — не требуем (выбор: маркер «запущено», не «закрыто»).
- Деплой покрывает ВЕСЬ диапазон коммитов (§4, ML-L6), не только HEAD.
- **Остаток: rebase.** Rebase переписывает SHA без прогонки commit-хуков → у переписанных
  коммитов нет ledger-записей → деплой блокирует; путь — `LADDER_SKIP=1` (audited) или избегать
  rebase уже-залесенных коммитов. Полное решение = серверный гейт Фазы 2.
- **Остаток: кросс-машинный ledger.** `logs/ladder_ledger/` локален (logs/ gitignored) — деплой
  с другой машины блокируется (как кросс-машинный baseline Фазы 1); Фаза 2.
- **Остаток: фиксы последнего прохода не видны предыдущему.** Chain-семантика честна: simplify
  не видел фиксов code-review; их ревьюит Codex-гейт. Fixpoint-итерация (гонять до «нет правок»)
  отвергнута: дорого, риск пинг-понга.

## Тестирование
- `scripts/ladder_gate.py` — юнит (изоляция: tmp git-репо): `tree`-вычисление (временный индекс,
  untracked включён, реальный индекс НЕ мутирован), **`begin`/`mark`-протокол (R6):** begin
  снимает before, mark после мутации пишет корректную пару before≠after; `mark` без `begin` →
  ошибка; **R7 анти-replay:** повторный `mark` после потребления pending → ошибка; `begin
  code-review` при tree ≠ `simplify.tree_after` → ошибка (ручная правка между проходами); **R6 код-пути:** `is_code_path(".githooks/pre-commit")` и `is_code_path("Makefile")`
  → True (коммит, ослабляющий хук/деплой-рецепт, НЕ exempt),
  `check-precommit`: **chain-валидность** (s.after==c.before && c.after==index → 0; разрыв
  цепочки — ручная правка между проходами → 2; протухший/нет → 2; **симулировать «/code-review
  внёс фиксы после /simplify» → цепочка валидна → 0** — ключевой R5-сценарий), non-code
  exempt (докс-коммит → 0 без маркеров), `LADDER_SKIP` (0+аудит), `ladder.enabled=false` →
  pre-commit пропуск (0).
- **check-ledger по диапазону (Codex R1/R2):**
  (а) **range-coverage** — промежуточный коммит БЕЗ ladder-записи в `baseline..HEAD` → блок,
      даже если у HEAD запись есть;
  (б) **tree-validation** — запись есть, но `ledger.tree != commit^{tree}` (протухшая/чужая) → блок;
  (в) **passes-complete** — не все КАНОНИЧЕСКИЕ проходы в записи → блок;
  (г) валидные записи (tree совпал, оба прохода) у ВСЕХ коммитов → 0;
  (д) **R5-кейсы:** exempt-noncode запись → покрыт; skipped-запись → покрыт + громкий аудит
      в выводе; коммит-предок `LADDER_EPOCH_SHA` без записи → покрыт (grandfathering);
      merge-коммит без записи → покрыт с пометкой; пост-эпоховый код-коммит без записи → блок.
- **flag-independence (Codex R1):** деплой-ladder-проверка блокирует даже при `ladder.enabled=false`
  (флаг гасит только pre-commit) — тест.
- **config-independence (Codex R3):** при `config.required_passes=[]` (или урезанном) деплой ВСЁ РАВНО
  требует канонический `DEPLOY_REQUIRED_PASSES`: коммит с ledger, где только один проход → блок,
  хотя config пуст. Доказывает, что урезание config не ослабляет деплой.
- `.githooks/pre-commit`/`post-commit` — smoke: коммит без маркеров abort; с маркерами ok +
  ledger записан с `tree==HEAD^{tree}`.
- `check-reviewed` — интеграция: коммит с валидным Codex-ledger, но без валидной ladder-записи
  диапазона → блок (лесенка обязательна независимо от Codex).

## Стоп-политика ревью этого дизайна — цикл ОСТАНОВЛЕН (R8)
Применена §«Стоп-политика цикла Codex-ревью (v2)». Траектория: R0 3×high → R2 2×high+1×med →
R3 1×high → R4 1×high(тест) → approve → **R5 свой повторный проход: 5 крупных дыр** (chain,
docs-exempt, grandfathering, skip-отравление, нереализуемый §5) → R5-Codex 2×high → R6 1×high →
R7 1×high (crash-окно, микроскопическое). **Стоп по хард-капу** (8 раундов > ~5): находки
сошлись от «дизайн неработоспособен» к асимптотически-мелким crash-окнам локального
гигиена-токена; все внесены. Следующий содержательный слой проверки — ревью РЕАЛЬНОГО кода
при реализации (per-task + Codex), а не девятый раунд по тексту.
Урок в стоп-политику: **крупный реворк (R5) легитимно переоткрывает цикл** — счётчик сухих
раундов сбрасывается на новой поверхности дизайна.
