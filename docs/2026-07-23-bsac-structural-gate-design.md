# Структурная валидация BSAC в design-маркере — дизайн

**Дата:** 2026-07-23. **Статус:** design (на Codex-ревью). **Тикет:** #2. **Tier:** 2
(design-гейт guard'ит качество ревью; провал — «отревьюен» стаб без сценарной матрицы).
**Опирается на:** file-режим маркера (тикет #3) — валидируем содержимое дизайн-ФАЙЛА.

## Проблема

`write-marker design --file` (тикет #3) записывает биндинг, но не проверяет, что дизайн
содержательный. Можно «отревьюить» трёхстрочный стаб без сценарной матрицы/BSAC — а skill
`design-review` шаг 2 МАНДАТИТ секцию BSAC (бизнес-сценарии + денежные краевые случаи). Гейт
эту мандату не энфорсит: дисциплина, а не механика.

## Что делаем

**Проверка BSAC — в `_marker_state`, ПЕРЕ-ВЫВОДИТСЯ из содержимого файла (не хранимый флаг,
Codex R3).** Биндинг валиден ⇔ `_hash_file(file)==hash` (нет дрейфа) И `_has_bsac(содержимое
файла)`. КЛЮЧЕВОЕ: когда hash совпал, файл == ОТРЕВЬЮЕННОМУ контенту (тот же байт-в-байт), значит
BSAC проверяется ровно на валидируемом контенте — исключает разъезд версий (хранимый `bsac` от
V2 при hash от V1 → откат к V1 давал бы ложный valid). Любой невалидный биндинг → маркер
`drifted` → G1 блок.

`add_design_file_binding` (file-режим) пишет биндинг `{file, hash: reviewed_hash}` (унификация с
#3: биндинг всегда пишется) и для НЕМЕДЛЕННОЙ обратной связи считает `_has_bsac` текущего файла;
exit 2, если hash≠текущий ИЛИ нет BSAC (совет оператору). Авторитетная проверка — в
`_marker_state` (пере-вывод), exit-код лишь информативный.

**Ключевое (Codex R1):** стаб-B, помеченный при валидном A, пишет биндинг в набор → `_marker_state`
пере-проверит B (нет BSAC) → маркер целиком `drifted` → G1 НЕ пропускает код B под маркером A
(coarse-marker fail-open закрыт — как #3 закрыл дрейф записью-и-инвалидацией, не reject-без-записи).

Escape — `--trivial` (существующий): для правок, которым BSAC не нужен (осознанно, в аудит).
Inline-режим структурно НЕ валидируется (нет файла-содержимого — ещё причина к file-режиму).

Совместно с дрейф-детектором (#3): BSAC и hash проверяются на ОДНОМ контенте; удалят секцию →
hash-дрейф И/ИЛИ пропажа BSAC → блок; добавят BSAC в стаб → hash-дрейф → надо перепометить.

Распознаваемые маркеры секции (регистронезависимо, любое ≥1 в файле):
`BSAC`, `бизнес-сценар`, `сценари`, `EARS`, `приёмочны`/`приемочны`, `acceptance`, `scenario`.
Лениво (любой из) — низкий false-positive; ложное срабатывание (структура есть, заголовок
нераспознан) escape'ится добавлением заголовка или `--trivial` (fail-safe трение).

## Scenario-first: пространство входов (ТЕКСТ до кода)

Измерения:
- **D1 содержимое файла:** {есть BSAC-заголовок, есть «сценарии», есть EARS, есть «приёмочные»,
  НЕТ ни одного распознаваемого}
- **D2 режим:** {file design, inline design, trivial}
- **D3 регистр/позиция:** заголовок в любом регистре, в любом месте файла

Сценарии (= тест-матрица):

| # | Условие | Ожидание |
|---|---|---|
| S1 | file, файл содержит `## BSAC` | биндинг записан (проверка пройдена) |
| S2 | file, содержит `## Сценарии`/`бизнес-сценарии` | записан |
| S3 | file, содержит `EARS` | записан |
| S4 | file, содержит `приёмочные критерии` | записан |
| S5 | file, НЕТ ни одного маркера (стаб) | биндинг записан + **exit 2**; `_marker_state` пере-вывод → drifted |
| S6 | file, маркер в другом регистре (`bsac`, `Scenario`) | валиден (регистронезависимо) |
| S7 | trivial | проверки НЕТ (escape для простых) |
| S8 | inline design (без --file) | структурной проверки НЕТ (нет файла; громкий inline-warning #3) |
| S9 | A валиден; пометка стаба B (без BSAC) | маркер (A+B) → **drifted** (B без BSAC) → G1 блок; НЕ остаётся валидным на A (Codex R1) |
| S10 | reviewed V1 без BSAC; при пометке файл=V2 с BSAC; потом откат к V1 | **drifted** — `_has_bsac(V1)=false` при совпавшем hash (Codex R3, анти-разъезд версий) |

Полнота: D1 покрыт; ключевые — стаб→невалидный маркер (S5), регистр (S6), escape trivial (S7),
стаб B рушит coarse-маркер (S9), пере-вывод исключает разъезд версий (S10).

Осознанно ВНЕ scope: «заголовок есть, но под ним пусто» (стаб с `## BSAC` без тела) —
эвристика не ловит; сознательное гейминг (как EMPIRICAL_SKIP) вне охвата; проверка ловит
СЛУЧАЙНЫЙ пропуск (забыл BSAC целиком) — §6a-класс. Семантическую полноту BSAC судит Codex на
ревью (это ратчет, не замена ревью).

## EARS — приёмочные критерии (→ тесты)

- **EARS-1:** `_marker_state` SHALL считать file-биндинг валидным ⇔ `_hash_file(file)==hash` И
  `_has_bsac(содержимое файла)` (≥1 распознаваемый маркер секции, регистронезависимо). BSAC
  пере-выводится из файла, НЕ хранится флагом — исключает разъезд hash/bsac по версиям. (S1-S4, S6, S10)
- **EARS-2:** `add_design_file_binding` SHALL записать биндинг `{file, hash: reviewed_hash}` и
  вернуть exit 2, IF hash≠текущий файл ИЛИ текущий файл не содержит BSAC (немедленный совет). (S5)
- **EARS-3:** Любой невалидный биндинг (дрейф ИЛИ нет BSAC) → маркер `drifted` → G1 блок; стаб-B
  при валидном A рушит coarse-маркер (не оставляет A разблокирующим, Codex R1). (S5, S9)
- **EARS-4:** WHERE режим `trivial` — структурная проверка SHALL НЕ применяться. (S7)

## BSAC — бизнес-сценарии и краевые случаи

- **BS-B1:** стаб-дизайн без сценарной матрицы нельзя пометить как отревьюенный (file-режим). (S5)
- **BS-B2:** простые правки не блокируются — `--trivial`. (S7)
- **ML-B1 [не денежный, качество]** §6a-класс: добавили поверхность без BSAC → если это НОВЫЙ
  дизайн-файл без секции → reject; если ДОПИСАЛИ в существующий отревьюенный файл → дрейф (#3).
  Вместе закрывают «стаб/неполный дизайн проходит как ревью».
- **ML-B2 [ОСТАТОК]** пустая секция под валидным заголовком / гейминг заголовком → эвристика не
  ловит; семантику судит Codex-ревью (ратчет, не замена). Escape `--trivial` аудируется.

## Actuator-safety (Tier 2)

- Монитор: reject виден (exit 2 + сообщение); `--trivial` в аудит.
- Escape: `--trivial` (осознанно). Ограниченные полномочия: проверка — на содержимом
  закоммиченного/reviewed файла.
- Recovery: гейт не мутирует внешнее состояние.

## Отвергнутые альтернативы

- **Warn вместо block:** беззубо; skill УЖЕ мандатит BSAC — механика энфорсит, `--trivial` = escape.
- **Требовать заголовок И непустое тело:** хрупкий парсинг «непустоты»; ловим случайный пропуск,
  семантику — Codex. Гейминг вне охвата (как SKIP).
- **Config-флаг `require_bsac`:** добавляет поверхность ослабления через конфиг; `--trivial`
  уже даёт per-mark escape. Фиксированные паттерны + trivial проще и без config-laundering.
- **Валидировать inline:** нет файла-содержимого; ещё причина к file-режиму.

## Тестирование

Юнит (tmp-файлы дизайна, как test_design_drift): S1-S9 через `add_design_file_binding`. Явно:
стаб пишет биндинг + exit 2, `_marker_state` пере-вывод → drifted (S5); разъезд версий V1/V2 (S10);
пометка стаба B при валидном A делает маркер (A+B) drifted → G1 блок, НЕ остаётся валидным на A
(S9, Codex R1); регистронезависимость (S6); trivial и inline не затронуты (S7/S8). Реальный
маршрут: `write-marker --file` стаба → exit 2 + gate_edit по код-пути → deny. NB: существующие
#3-тесты используют дизайн-контент с распознаваемым заголовком (иначе bsac=false — стаб).
