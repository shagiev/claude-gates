# Детектор дрейфа design-review (маркер↔файл) — дизайн

**Дата:** 2026-07-23. **Статус:** design (на Codex-ревью). **Тикет:** #3. **Tier:** 2
(actuator-safety — design-гейт guard'ит непроверенную actuator/data-loss поверхность; провал
в fail-open = необратимая правка идёт мимо ревью).

## Проблема

`write-marker design <hash>` записывает хэш отревьюенного дизайна, но плагин его НИКОГДА не
перепроверяет — маркер разблокирует правки кода на всю сессию (`has_marker` смотрит только:
сессия совпала, kind=design, hash непустой). Отсюда дыра: отревьюил дизайн v1 → пометил
hash(v1) → дописал §6a (v2) → маркер всё ещё hash(v1), G1 пропускает, новая поверхность
независимого ревью НЕ проходит. В бою §6a была про необратимое удаление данных БД (data-loss).

## Что делаем

Маркер `design` в file-режиме держит **НАБОР** биндингов `designs: [{file, hash}]`, где `hash`
— это хэш РЕЗУЛЬТАТА ревью (переданный оператором из отревьюенного текста), а не «что сейчас в
файле». `has_marker` (зовётся из G1 на каждой правке кода) валиден ТОЛЬКО если сессия совпала
И КАЖДЫЙ биндинг: файл читается и его sha256 == записанному `hash`. Дрейф любого (файл изменён/
удалён/непрочитан) → маркер НЕВАЛИДЕН → G1 блокирует правку кода «дизайн изменился — /design-review».

**Reviewed-hash биндинг (R1-F1, анти-fail-open).** `write-marker` НЕ доверяет текущему файлу:
он принимает ОЖИДАЕМЫЙ reviewed-хэш (из текста, отправленного на ревью) и **отклоняет**, если
sha256 текущего файла ему не равен. Так «добавили §6a между ревью и пометкой» ловится на самой
пометке: reviewed-hash от v1, файл уже v2 → mismatch → reject, маркер не пишется.

**Набор, не единственный файл (R1-F2).** `write-marker --file` ДОПОЛНЯЕТ/обновляет запись
этого файла в `designs`, не затирая прочие. Пометил дизайн A, затем B → в наборе оба → правка
A ПОСЛЕ пометки B → дрейф A → G1 блокирует (маркер разблокирует все код-пути, потому covered
должны быть ВСЕ отревьюенные дизайны сессии).

Точка энфорсмента — G1 (PreToolUse на правках кода): дёшево (чтение файлов + sha256), ловит
§6a в момент имплементации новой поверхности. Дизайн-файлы — в `docs/` (не код, G1 на них не
срабатывает), но СЛЕДУЮЩАЯ правка кода перепроверяет весь набор.

Режимы `write-marker design`:
- **file-режим (новый):** `write-marker design "<detail>" <reviewed_hash> --file <path>` — гейт
  читает файл, sha256; если ≠ `reviewed_hash` → **reject** (файл изменился с ревью). Иначе
  добавляет `{file, hash: reviewed_hash}` в `designs` (merge в маркер сессии).
- **inline-режим (легаси, back-compat):** `write-marker design "<detail>" <hash>` (без `--file`)
  — как сейчас, `designs` нет. Дрейф-проверка невозможна (нет источника-файла) → ведёт себя как
  раньше. Скилл толкает к file-режиму для нетривиальных дизайнов.

## Scenario-first: пространство входов (ТЕКСТ до кода)

Измерения:
- **D1 маркер:** {отсутствует, inline(legacy, без `designs`), file(с `designs`)}
- **D2 биндинги набора (file):** {все совпали, один дрейфнул, один отсутствует/непрочитан}
- **D3 сессия:** {совпадает, чужая, пустая}
- **D4 kind:** {design, trivial}
- **D5 write-marker --file:** {reviewed_hash == файл, reviewed_hash ≠ файл}
- **D6 набор:** {один дизайн, несколько (A,B)}

Сценарии (= тест-матрица):

| # | Условие | Ожидание |
|---|---|---|
| S1 | маркера нет | `absent` → G1 deny (существующее) |
| S2 | inline design (legacy, без `designs`), сессия ✓, hash непустой | `valid` → allow (back-compat) |
| S3 | file design, все биндинги СОВПАЛИ | `valid` → allow |
| S4 | file design, один биндинг ДРЕЙФНУЛ | `drifted` → **deny** «дизайн изменился» (ФИКС) |
| S5 | file design, файл биндинга ОТСУТСТВУЕТ | `drifted` → **deny** (дизайн исчез = ре-ревью) |
| S6 | file design, чужая сессия | `foreign` → deny (session-проверка ДО дрейф-проверки) |
| S7 | trivial (файла нет by design) | `valid` → allow (trivial дрейф-проверке не подлежит) |
| S8 | `write-marker --file`, reviewed_hash == текущий файл | пишет биндинг `{file, reviewed_hash}` в набор |
| S8b | `write-marker --file`, reviewed_hash ≠ текущий файл | **REJECT** (файл изменился с ревью), маркер НЕ пишется (R1-F1) |
| S9 | дизайн дрейфнул, затем пере-`write-marker` (reviewed_hash от новой версии) | `valid` → allow (легитимный ре-ревью) |
| S10 | файл биндинга не читается (OSError) | `drifted` → deny (fail-closed, не «валиден») |
| S11 | пометил A, затем B; правка A ПОСЛЕ | `drifted` (A в наборе) → **deny** (R1-F2, мульти-дизайн) |

Полнота: D1–D6 покрыты; ключевые — inline-back-compat (S2) vs file-drift (S4), reviewed-hash
reject (S8b), мульти-дизайн (S11), missing/OSError fail-closed (S5/S10), session-до-дрейфа (S6),
re-mark сбрасывает (S9).

Осознанно ВНЕ scope: inline-маркеры дрейф-проверить нельзя (нет источника) — легаси, скилл
толкает к file-режиму; кросс-сессионный дрейф (маркер пер-сессионный, сбрасывается на
SessionStart) ловит деплой-гейт + конституция (safety-net); ручная правка самого маркер-файла
(.claude/, gitignored) = эквивалент CODEX_REVIEW_SKIP (осознанный обход, вне гейта).

## EARS — приёмочные критерии (→ тесты)

- **EARS-1:** WHEN design-маркер содержит `designs` — `has_marker` SHALL пересчитать sha256
  КАЖДОГО файла набора и вернуть valid ТОЛЬКО если все совпали с записанными `hash`. (S3)
- **EARS-2:** IF любой файл набора отсутствует, не читается ИЛИ его хэш ≠ записанному (дрейф) —
  THEN маркер SHALL быть невалиден, а G1 SHALL заблокировать правку кода с ре-ревью-сообщением. (S4, S5, S10, S11)
- **EARS-3:** WHERE design-маркер БЕЗ `designs` (легаси inline) — `has_marker` SHALL вести себя
  как прежде (дрейф-проверка невозможна, back-compat). (S2)
- **EARS-4:** WHEN `write-marker` дан `<reviewed_hash> --file <path>` — гейт SHALL вычислить
  sha256 текущего файла и, ЕСЛИ он ≠ `reviewed_hash`, ОТКЛОНИТЬ запись (файл изменился с ревью,
  fail-closed); иначе записать биндинг `{file, hash: reviewed_hash}`. Хэш — из результата
  ревью, не «что сейчас в файле». (S8, S8b)
- **EARS-5:** `write-marker --file` SHALL ДОПОЛНЯТЬ/обновлять биндинг файла в наборе сессии, не
  затирая прочие (несколько отревьюенных дизайнов — все под дрейф-проверкой). (S11)
- **EARS-6:** IF файл биндинга не читается (OSError) — THEN дрейф-проверка SHALL трактовать это
  как дрейф (deny), не как valid (fail-closed). (S10)
- **EARS-7:** Session-проверка SHALL идти ДО дрейф-проверки (чужая сессия → foreign вне
  зависимости от дрейфа). (S6)

## BSAC — бизнес-сценарии и денежные краевые случаи

Бизнес-сценарии:
- **BS-D1:** правка дизайна после `write-marker` (file-режим) → следующая правка кода
  блокируется до ре-ревью. (S4)
- **BS-D2:** back-compat — inline-маркеры (легаси) не ломаются. (S2)

Денежные/fail-open краевые случаи:
- **ML-D1** пост-ревью добавление destructive/actuator/data-loss поверхности (§6a-инцидент) →
  дрейф → блок следующей код-правки (S4/S5). Плюс safety-net: деплой-гейт + конституция
  (destructive = blocking) ловят на уровне кода независимо.
- **ML-D2** дизайн-файл не читается → трактуется как valid → непроверенная поверхность. →
  EARS-6 fail-closed (OSError = дрейф).
- **ML-D5 [R1-F1]** §6a добавили в файл МЕЖДУ ревью и `write-marker` → пометка заверила бы v2.
  → reviewed_hash от отревьюенного текста; `write-marker` reject при sha256(файл)≠reviewed_hash.
- **ML-D6 [R1-F2]** правка ранее отревьюенного дизайна A после пометки B → escape. → набор
  биндингов, valid только если ВСЕ совпали; правка A → дрейф → deny.
- **ML-D3 [ОСТАТОК]** inline-маркер дрейф-проверить нельзя (нет источника). → скилл требует
  file-режим для нетривиальных; inline остаётся для `--trivial`-класса и легаси.
- **ML-D4 [ОСТАТОК]** ручная правка маркер-файла под дрейфнувший хэш = осознанный обход
  (эквивалент SKIP) — вне гейта Фазы 1.

## Actuator-safety (Tier 2)

- **Монитор исхода:** дрейф виден в G1-deny + (файл-режим) в git-диффе дизайн-файла.
- **Kill-switch/escape:** ре-ревью + пере-`write-marker` (легитимный путь); аварийно — как
  G1 в целом fail-open, но дрейф — это KNOWN-bad (не availability), потому deny, не allow.
- **Ограниченные полномочия:** маркер пер-сессионный, хэш от файла (не от аргумента).
- **Recovery:** гейт не мутирует внешнее состояние.

Определение готово: реальный маршрут воспроизведён — правка дизайн-файла после `write-marker`
РЕАЛЬНО блокирует следующую G1-правку кода (не только юнит `has_marker`).

## Отвергнутые альтернативы

- **Warn вместо block:** беззубо; block — смысл фичи (дрейф = known-bad).
- **Проверять дрейф только на деплое:** позже; G1 ловит в момент правки. Деплой-гейт остаётся
  как cross-session safety-net (не замена).
- **Неявно трекать «дизайн» без файла:** хрупко; явный file-биндинг чист и виден в git.
- **Per-design-file G1 (маркер на список код-путей дизайна):** большой реворк UX G1, Фаза 2;
  file-дрейф даёт 80% пользы малой ценой.
